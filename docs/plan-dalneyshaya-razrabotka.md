# План дальнейшего создания: дашборд группы + контур данных

*Актуально на момент прототипа: веб-приложение `web/` с мок-данными по юрлицам.*

---

## 0. Что уже есть

- **Фронтенд:** Vite + React + TS, Tailwind, React Router, Recharts; сводка по группе, список компаний, карточка юрлица.
- **Данные:** файл `web/src/data/companies.ts` — мок под UI (не источник правды).
- **Материалы:** в `docs/neral-vault/` — структура группы, идеи метрик, концепт TG-дашборда (как целевая картина, не обязательно следующий шаг).

---

## 1. Зафиксировать цель и границы (1–3 дня, без кода)

Согласовать с заказчиком / гендиректором:

| Вопрос | Зачем |
|--------|--------|
| Какие **юрлица** в первой волне (MVP) | Объём интеграций |
| Какие **5–10 метрик** обязательны на главном экране | Приоритет разработки |
| **Источник правды** (1С редакция, СБИС, Excel, вручную) | Архитектура загрузки |
| **Доступ:** только внутри сети / VPN / публичный с логином | Безопасность и хостинг |
| Нужен ли **параллельно** Telegram или только веб на старте | Разделение работ |

Результат: короткое ТЗ в 1–2 страницы или список в задаче — от него зависят фазы 2–3.

---

## 2. Бэкенд и контракт API (1–2 недели)

**Цель:** фронт читает данные не из мока, а из вашего API.

1. Выбрать стек бэкенда (например **Node Fastify** или **Python FastAPI**) — один сервис `api/`.
2. Описать **OpenAPI** (или хотя бы таблицу эндпоинтов):  
   `GET /companies`, `GET /companies/:id`, `GET /group/summary`, позже `GET /metrics?company=&period=`.
3. Реализовать **чтение из того, что есть сейчас**:  
   - вариант A: CSV/Excel по расписанию;  
   - вариант B: выгрузка из 1С (OData / HTTP-сервис / файл);  
   - вариант C: ручное обновление через простую админ-форму (временный костыль).
4. Подключить фронт: переменная окружения `VITE_API_URL`, замена импорта мока на `fetch`.

**Критерий готовности:** сводка и карточки наполняются из API; мок можно оставить как fallback для разработки.

---

## 3. Углубление дашборда (2–4 недели, итерациями)

По приоритету из ТЗ:

1. **Фильтры:** период (год/квартал), валюта только ₽ на старте.
2. **Дополнительные виджеты:** дебиторка, кредиторка, остатки ДС — если данные появляются в API.
3. **Таблицы и экспорт** (CSV/PDF) для финблока — по запросу.
4. **Роли и вход:** простая авторизация (логин/пароль или SSO позже), разделение «видит всю группу» / «только свою фирму».

**Критерий:** пользователи из пилотной группы ежедневно смотрят актуальные цифры в вебе.

---

## 4. Надёжность и эксплуатация

1. **Логирование** запросов API и ошибок загрузки из 1С.
2. **Резервное копирование** конфигов и БД (если появится БД).
3. **CI:** `npm run build` + линтер на каждый push (GitHub Actions / GitLab CI — как заведёте репозиторий).
4. **Сборка Docker** для `web` + `api` — удобно для OVH и одинакового окружения.

---

## 5. Telegram и MAX (диалог с сотрудниками)

**Решения:** сразу **диалог**; только **сотрудники** (allowlist); **один и тот же** текст ответа в оба канала (см. [`docs/integraciya-telegram-i-max.md`](./integraciya-telegram-i-max.md)).

1. Модуль **`api/bots`**: webhook Telegram + MAX → общий роутер команд → те же отчёты, что для веба → `composeReply()` → адаптеры TG/MAX.  
2. Реестр **`messenger_allowlist`**: привязка внешнего id к сотруднику и доступным юрлицам.  
3. **Push/дайджест** (пороги, cron) — поверх того же текста ответа, можно второй очередью.

**Критерий:** сотрудник из списка получил в TG или MAX корректный ответ на тестовую команду по живым или пилотным данным.

## 6. Развёртывание (OVH1 или аналог)

Из ваших материалов — отдельный контур **OVH1**.

1. ВМ или контейнеры: `api` + статика `web` (nginx) или один хост.
2. HTTPS, домен, ограничение по IP/VPN при необходимости.
3. Секреты (пароли 1С, токены) — в env / vault, не в git.

---

## 7. ИИ-слой (после дашборда и данных)

Не начинать раньше стабильного **контура цифр** и **регламентов доступа**.

1. **RAG** по документам (договоры, приказы) — отдельное хранилище, индексация, ответы только со ссылкой на источник.
2. **NL → SQL или готовый отчёт** — только whitelisted запросы / шаблоны, чтобы снизить галлюцинации по деньгам.
3. **Голос** — опционально, после веба и алертов.

---

## 8. Порядок работ (рекомендуемая очередь)

```text
ТЗ и метрики (фаза 1)
    → API + первая выгрузка (фаза 2)
        → Дашборд web + параллельно bots TG/MAX (диалог, allowlist) (фаза 2b–3)
            → Push/дайджест по порогам (фаза 5)
                → Деплой прод (фаза 6)
                    → ИИ / RAG (фаза 7)
Параллельно: надёжность и Docker (фаза 4)
```

---

## 9. Риски

| Риск | Что делать |
|------|------------|
| Разный учёт в юрлицах | Начать с 1–2 фирм и одного набора полей |
| Нет выделенного владельца данных со стороны заказчика | Назначить до старта фазы 2 |
| «Сразу ИИ» | Удержать фокус: сначала правильные цифры на экране |

---

## 10. Следующий конкретный шаг

1. Заполнить таблицу из **раздела 1** (хотя бы устно в чат/звонок и зафиксировать в этом файле).  
2. Создать репозиторий `api/` с одним эндпоинтом `GET /companies`, отдающим то же, что сейчас в моке.  
3. Переключить `web` на этот API локально.

После этого план уточняется по срокам в **человеко-днях** под выбранный стек и объём 1С.
