# Интеграция дашборда с Telegram и MAX

Цель: **одни и те же данные и правила**, что и у веб-дашборда — плюс **диалог** в мессенджерах (вопрос → ответ по цифрам/спискам в рамках доступа).

---

## Зафиксированные решения (заказчик)

| # | Решение |
|---|--------|
| 1 | **Сразу диалог** — пользователь пишет боту, бэкенд отвечает (не откладываем только на «push потом»). Push/дайджест можно добавить тем же движком. |
| 2 | **Только сотрудники** — бот не для внешнего рынка; доступ по **allowlist** (список людей/идентификаторов), привязка к роли/юрлицу из учёта или HR. |
| 3 | **Один и тот же текст ответа** — одна функция формирует тело сообщения; адаптеры **Telegram** и **MAX** получают **одинаковую строку** (и при необходимости одни и те же кнопки-ссылки, если платформа позволяет). Если у человека подключён только один канал — уходит одна копия. |

---

## Архитектура под диалог (один бэкенд)

```text
                    ┌── webhook / long poll ──► Telegram
                    │
[Пользователь] ────►│  единый слой «Диалог»
                    │       ↓
                    │  (проверка allowlist, лимиты, аудит)
                    │       ↓
                    │  тот же сервис метрик/справочников,
                    │  что и у веб-дашборда
                    │       ↓
                    │  composeReply() → одна строка текста
                    │       ↓
                    └── MAX Bot API ──► MAX
```

- **`web/`** — только UI; **токены и секреты** только на сервере.
- Входящее сообщение → **идентификация пользователя** (Telegram `user_id` / MAX id) → **сопоставление с сотрудником** (таблица `messengers_accounts`: канал, внешний id, employee_id, доступные юрлица).
- Ответ строится **один раз** → в **оба** адаптера с тем же текстом (если для сотрудника включены оба канала — редкий кейс; чаще один канал на человека — текст всё равно единый по шаблону).

---

## Диалог: с чего начать в логике (MVP)

1. **Команды по шаблону** (быстро и предсказуемо): например `/start`, `/help`, `/cash`, `/debtors`, `/company` — парсинг без «свободного ИИ» на первом этапе.  
2. **Свободный текст** можно добавить вторым этапом через LLM **только** как перевод «вопрос → вызов тех же отчётов», с whitelist и логами.  
3. **Лимиты:** не более N запросов в минуту на пользователя; таймаут отчёта; при ошибке — короткое «не смог, обратитесь в IT».

Так вы получаете «диалог» с первого дня, но безопаснее, чем сразу голый LLM на деньгах.

## Telegram

- [Bot API](https://core.telegram.org/bots/api): `sendMessage`, webhook или long polling.  
- Allowlist: **только** `chat_id` из реестра сотрудников.  
- Для сотрудников: удобна **личка с ботом** или закрытая группа с ботом (политика доступа — отдельно).

---

## MAX (мессенджер VK)

- [dev.max.ru/docs](https://dev.max.ru/docs), библиотека `@maxhub/max-bot-api`.  
- **Только сотрудники:** тот же allowlist, отдельная колонка MAX user id.  
- Прод: учёт **верификации организации / модерации** бота на стороне MAX.  
- Ответы — **тот же текст**, что сформирован для Telegram (после экранирования под лимиты MAX, если длина/формат отличаются — обрезка/префикс, но смысл один).

## Безопасность (усиленно под диалог)

1. **Без allowlist — нет ответа** (даже на `/start` можно отвечать нейтрально «нет доступа»).  
2. **Привязка к сотруднику и scope юрлиц** — в ответе не показывать чужие компании.  
3. Секреты в **env** / vault; логировать **факт запроса** и тип команды, не логировать полные тексты переписки без необходимости.  
4. **Аудит:** кто спросил, какой отчёт отдался (для финансовых вопросов).  
5. Webhook: секретный URL, проверка подписи Telegram; для MAX — по их актуальной доке.

---

## Связка с планом разработки

1. Поднять **`api/`** с данными для дашборда и **одним** внутренним модулем `reports` (функции «касса», «дебиторка» — те же, что позже дернет и веб).  
2. Модуль **`bots`**: два входа (TG webhook, MAX), общий **роутер команд** → `reports` → `composeReply()`.  
3. Таблица/файл **`messenger_allowlist`** + админка или ручное заполнение на пилоте.  
4. Веб остаётся параллельно; позже — общие push-правила поверх того же `composeReply`.

## Переменные окружения (обновлённый черновик)

| Переменная | Назначение |
|------------|------------|
| `TELEGRAM_BOT_TOKEN` | токен бота |
| `TELEGRAM_WEBHOOK_SECRET` | секрет пути или заголовок |
| `MAX_BOT_TOKEN` | токен бота MAX |
| `PUBLIC_BASE_URL` | HTTPS для webhook Telegram |
| `DIALOG_RATE_LIMIT_PER_MIN` | лимит запросов на пользователя |

---

## Следующий шаг в разработке

1. Описать **список команд MVP** (3–7 штук) и какие отчёты за ними стоят.  
2. Завести **`api/`** + заглушки webhook TG/MAX, которые на allowlist отвечают тестовой строкой.  
3. Подключить **первый реальный отчёт** (например сводка по одной фирме) — один текст → оба адаптера.

После этого можно подключать фронт `web/` к тому же `api/`.
