# Executive TG-дашборд для ИД Группы НЕРАЛ — концепция

> Запрос клиента: real-time мониторинг, триггерные алерты на отклонения, голосовые запросы → краткая сводка/диаграммы/отчёты. Базис — [[company-snapshot]] + [[related-companies]] + [[ai-benchmarks]]. Инициатива №1 из [[automation-roadmap]].

## 1. Концепция в одной строке

«WhatsApp/Telegram-бот, который для исполнительного директора в любой момент времени отвечает на вопрос „как идут дела?" — конкретными цифрами, диаграммами и алертами по всем 35 предприятиям группы».

## 2. Top-10 метрик на главный экран

Подбор сделан под профиль агро/FMCG-холдинга. Каждой метрике приведён источник данных и обоснование.

| # | Метрика | Источник | Зачем |
|---:|---|---|---|
| 1 | **Касса по группе (cash on hand) на сегодня** | 1С: банк-клиент / выписки | Базовое здоровье. Триггер кэш-разрыва — №1 риск для холдинга. |
| 2 | **Дебиторка > 60 дней (AR aging)** | 1С: продажи / СБИС | Кто и сколько должен. Прокси для управления оборотом. |
| 3 | **Кредиторка к погашению на 7 дней вперёд** | 1С: закупки | Чтобы не словить просрочку поставщикам/банкам. |
| 4 | **Дневной выпуск по ключевым SKU (тонн, %от плана)** | МES / 1С:УПП БМК, Башхолод | Прямая операционная метрика для FMCG-части. |
| 5 | **Состояние парка с/х техники (готовность %)** | Телематика (Агросигнал/OBD) | В пик сезона критично. Должно быть >90%. |
| 6 | **Прогноз погоды vs план работ (окно посева/уборки, дни)** | Open-Meteo + GIS-поля | Прямой контроль рисков. |
| 7 | **NDVI и yield-прогноз по ключевым полям** | Sentinel-2 + ML-модель | Качественный контроль растениеводства. |
| 8 | **Цены закупки молока + цены реализации сыра/мороженого (рынок)** | Союзмолоко, IndexBox, парсеры | Маржинальность переработки в реальном времени. |
| 9 | **Топ-5 контрактов: статус исполнения** | 1С / СБИС / госконтракты | Чтобы не словить штраф за просрочку. |
| 10 | **Численность персонала vs план (по юрлицам)** | 1С:ЗУП | Контроль ФОТ и кадровых дыр в пик сезона. |

## 3. Top-10 триггеров (push-алерты в TG ИД-у)

Триггер = условие → push-сообщение + (optionally) кнопка «детали». Список — приоритезированный.

1. **Кэш-разрыв 30 дней:** прогнозируемый остаток денег < платежей в горизонте 30 дней. → «Алерт: к 15 числу cash gap –12 млн ₽».
2. **Просроченная дебиторка > X млн ₽:** AR-aging buckets >60 / >90 пересекают порог. → «3 контрагента в зоне 90+ дней, всего 8,4 млн ₽».
3. **Топливо +15% MoM на любой СХП:** аномалия в ГСМ-расходах (часто = воровство или поломка). → «СХП Чишмы: ГСМ +18% к прошлому месяцу».
4. **Срыв окна посева/уборки:** прогноз дождя <3 дня + поле не убрано/не засеяно. → «Поле №7 Буздяк: 4 дня окна, не убрано 240 га».
5. **Простой техники в пиковый сезон:** комбайн/трактор offline >12 ч в окне посева/уборки. → «Комбайн КЗ-12 простаивает 14 ч, поле №3».
6. **Падение надоев >5% на стадо:** отклонение от 30-дневной средней. → «Стадо Чишмы: надой –6,2% за 3 дня — проверить корма/здоровье».
7. **Просрочка по госконтракту/коммерческому контракту:** срок исполнения через <7 дней, прогресс <80%. |
8. **Контрагент-поставщик задерживает поставку:** ключевой поставщик X дней просрочки. → «Поставщик упаковки — задержка 5 дней, риск остановки линии БМК». |
9. **Резкое изменение биржевой цены молока/зерна >5% за неделю:** триггер пересмотра коммерческой политики. |
10. **Внеплановое арбитражное дело > Y млн ₽:** мониторинг kad.arbitr.ru на новые иски против юрлиц группы. |

## 4. Top-10 голосовых запросов (NL → ответ)

Каждый запрос: что нужно от данных, какой формат ответа (text / chart / table). Запросы на русском, как директор сказал бы.

| # | Запрос | Источник данных | Формат ответа |
|---|---|---|---|
| 1 | «Сколько у нас денег на счетах сейчас?» | 1С банк, агрегация по юрлицам | Text: «1 254 млн ₽ совокупно. Топ-3 счёта: ...» |
| 2 | «Какой кассовый разрыв на месяц вперёд?» | 1С + CF-модель | Text + line chart остатков по дням |
| 3 | «Сколько мы должны поставщикам сейчас и кому больше всего?» | 1С AP | Table: top-10 поставщиков, сумма, просрочка |
| 4 | «Что с урожайностью на поле №3 Чишмы?» | Sentinel + ML | Map+chart: NDVI динамика + прогноз tons |
| 5 | «Сколько молока надоили вчера и за неделю?» | МES молочной фермы / 1С | Text + bar chart по дням |
| 6 | «Покажи топ-5 контрактов под риском просрочки» | 1С / СБИС | Table: контрагент, срок, % исполнения, рисковый комментарий |
| 7 | «Какая выручка по Башкирскому холоду за апрель vs план?» | 1С УТ БХ | Number + waterfall до плана |
| 8 | «Где сейчас комбайн КЗ-12 и что с ним?» | Телематика | Map + статус (онлайн/ошибки) |
| 9 | «Какие риски ты видишь на эту неделю?» | Все триггеры + LLM-саммари | Text: summary 5–7 пунктов |
| 10 | «Сравни рентабельность сестринских СХП по итогам квартала» | 1С + расчёты | Table: revenue, EBITDA, EBITDA% по Чишмы/Буздяк |

Voice→text — Whisper или Yandex SpeechKit (последний предпочтительнее для русского и работает on-prem).

**Ключевые компоненты:**

1. **TG-bot** (aiogram + FastAPI backend) — единая точка входа. Один бот на all functions.
2. **STT слой** — Whisper (open-source) или Yandex SpeechKit (для русского точнее, on-prem доступен).
3. **LLM** — два варианта:
   - Cloud: GPT-4 / Claude — быстрее MVP, но данные уходят в чужой контур.
   - On-prem: Qwen2.5 32B / YandexGPT / Saiga (русский fine-tune Mistral) — медленнее MVP, но данные группы остаются внутри. **Рекомендуется on-prem** для группы такого масштаба.
4. **Vector DB** — pgvector в PostgreSQL (без отдельного зоопарка). Туда грузим: справочники юрлиц, договоры, регламенты, исторические протоколы.
5. **Data Warehouse** — ClickHouse (быстрый для агрегаций, бесплатный, открытый). Альтернатива — Greenplum.
6. **ETL** — Airflow + dbt. Из 1С через REST/OData или ODBC, из СБИС — API, из телематики — MQTT.
7. **Tool-use на LLM:** агент с тремя инструментами:
   - `sql_query(question, schema)` — генерация и выполнение SQL по DWH.
   - `semantic_search(query)` — поиск по vector DB (договоры, протоколы).
   - `make_chart(data, type)` — построение диаграммы (Plotly → PNG).
8. **Scheduler** — отдельный сервис (Prefect/APScheduler) для триггеров. Запускает SQL-условия → если порог пробит → push в TG.

**Безопасность:**
- Allowlist Telegram ID-шников. Никаких публичных команд.
- LLM запросы логируются (compliance).
- On-prem LLM — данные группы не покидают периметр.
- Раздельные роли: ИД — full access; финдир — все финансы; директор СХП — только своя сестра.

## 6. Что НЕ делать в MVP

- Не строить кастомную ERP — берём данные из существующих 1С.
- Не агрегировать всю группу сразу — MVP на 2 юрлица (Агроцентр + Чишмы), потом масштабируем.
- Не закладывать speech synthesis (TTS) — звуковой ответ ИД-у не нужен, текст+картинка достаточно.
- Не делать сложную RBAC на старте — allowlist+простые роли.

## 7. Открытые вопросы к клиенту

1. Какая ERP стоит у группы? 1С (какая редакция?), SAP, что-то самописное?
2. Есть ли уже централизованный data lake/warehouse, или каждая сестра — silo?
3. Какие компании клиент хочет в MVP-скоупе? (Рекомендация: Агроцентр + СХП Чишмы.)
4. Кто список получателей дашборда: только ИД, или круг шире (финдир, директор СХП)?
5. Готовность развернуть LLM on-prem (нужны GPU-серверы) или cloud LLM приемлем?
6. Парк техники: какая модель, есть ли уже какая-то телематика?
